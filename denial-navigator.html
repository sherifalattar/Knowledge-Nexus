Structural grammar

Code anatomy, without the fog machine

A practical reference architecture. It helps users read codes faster, even when the exact code book differs between payers, TPAs, regulators, or EDI standards. Most systems use a compact syntax: an alphabetic prefix classifies the family, a numeric value identifies the reason, and an optional suffix carries severity, routing, appealability, or document requirements.

PREFIX
Category / domain

Examples: AUTH for authorization, ELIG for eligibility, NCOV for non-covered, DUPL for duplicate, DOC for missing documentation.

000–999
Specific reason

The number narrows the issue: no prior approval, inactive policy, excluded benefit, coding mismatch, duplicate submission, or clinical insufficiency.

.A / -R
Optional modifier

A local layer that may indicate appeal path, resubmission requirement, documentation group, claim line, severity, or edit version.

Critical distinction

Global structure is not a global code book

There is no single denial-code dictionary used by every healthcare market. What is globally recognizable is the logic: short alphanumeric codes that compress a denial reason, responsible party, remittance context, or next action into a readable identifier.

Navigation layer

Common denial-code families

These families work as a clean navigation layer in a denial lookup card or executive dashboard.

AUTHAuthorization

Prior approval, referral, extension, validity period, level of care, or approval scope.

ELIGEligibility

Member coverage, policy status, waiting period, enrollment, or network entitlement.

NCOVNon-covered

Benefit exclusion, service limitation, plan restriction, or policy non-entitlement.

DOCDocumentation

Missing clinical notes, reports, investigation results, consent, or operative details.

DUPLDuplicate

Repeated authorization, same date/service/member, claim overlap, or prior processed request.

X12 / remittance context

Where formal standards fit

In formal remittance environments, denial and adjustment explanation often uses recognized code sets rather than local dashboard labels.

Reason + responsibility

CARC + Group Code

Claim Adjustment Reason Codes explain why a line was paid differently than billed. Group Codes are alphabetic responsibility indicators — contractual obligation, patient responsibility, or payer initiation.
